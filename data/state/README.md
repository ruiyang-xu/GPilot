## Purpose

The JSON state layer is the canonical machine-readable data store for the Fund OS. JSON files replace xlsx as the primary source of truth for programmatic operations while keeping xlsx as human-readable exports and reference views.

## Design Principles

### 1. **JSON as Canonical State**
- JSON files are the authoritative machine-readable state
- All Claude Code commands and scheduled tasks read/write to JSON first
- JSON is version-controllable (diffable, mergeable) unlike binary xlsx
- Each file is an array of objects with consistent schema (see `schema.json`)

### 2. **XLSX as Human-Readable Export**
- Excel workbooks in `../data/` are derived views of JSON state
- Regenerated by the `sync-state-to-xlsx` command on-demand or scheduled
- Humans can view/edit xlsx locally for offline work
- Changes to xlsx must be re-synced back to JSON (manual or automated)

### 3. **Atomic Updates**
- JSON supports partial field updates without rewriting entire file
- Commands can update specific fields (e.g., deal status) without touching others
- Reduces merge conflicts and improves concurrency

### 4. **Git-Friendly**
- JSON diffs are human-readable and reviewable
- Merge conflicts are easier to resolve than in binary xlsx
- State changes are auditable via git history

## File Descriptions

### `deals.json`
Deal opportunity pipeline tracking. Array of deal objects.

**Usage**: Commands like `/deal-screen`, weekly pipeline reviews, and sourcing workflows.

**Fields**:
- `deal_id`: Unique identifier (e.g., "deal-acme-20260403")
- `company`: Company name
- `sector`: Primary sector (SaaS, AI/ML, Fintech, etc.)
- `stage`: Current funding stage (Pre-seed, Seed, Series A/B/C/D, Growth, Public)
- `status`: Deal status (Sourced, Screening, DD, Term Sheet, Invested, Passed, Archived)
- `score`: Internal investment score (0-100)
- `entry_valuation_usd`: Entry valuation if known
- `check_size_usd`: Target check size
- `check_size_pct_ownership`: Target ownership %
- `source`: How deal was sourced (Outbound, Inbound, Warm intro, etc.)
- `assignee`: Team member leading this deal
- `screened_date`: When screening was completed (ISO 8601)
- `next_action`: Next step (e.g., "schedule founder call", "request financials")
- `next_action_date`: When next action is due
- `notes_path`: Path to detailed notes.md in `/deals/{company}/`
- `created_date`: When deal was added to pipeline
- `last_updated`: Last modification date

### `portfolio.json`
Active portfolio company data. Array of portfolio company objects.

**Usage**: Portfolio reviews, KPI tracking, board prep, valuation updates.

**Fields**:
- `portfolio_id`: Unique identifier (e.g., "port-quantum-labs")
- `company`: Company name
- `entity_name`: Legal entity name
- `sector`: Sector classification
- `stage_at_investment`: Stage when we invested
- `initial_investment_date`: Date of initial investment
- `initial_check_usd`: First check size
- `initial_ownership_pct`: First ownership stake
- `current_valuation_usd`: Latest valuation (post-money)
- `valuation_source`: How valuation was determined (DCF, comps, 409A, etc.)
- `valuation_confidence`: Confidence level (High, Medium, Low)
- `last_valuation_date`: When valuation was last updated
- `last_kpi_update_date`: When KPIs were last reviewed
- `board_seat`: Whether we have board representation
- `next_board_date`: Date of next board meeting
- `runway_months`: Estimated months of cash runway
- `status`: Current status (Active, Follow-on needed, Monitoring, Exiting, Exited)
- `health_score`: Overall portfolio company health (1-10)
- `key_risks`: Critical risks (string, pipe-separated)
- `notes_path`: Path to detailed notes in `/deals/{company}/`
- `last_updated`: Last modification date

- `wechat_formatted_path`: Path to WeChat-formatted version (CN)
- `linkedin_formatted_path`: Path to LinkedIn-formatted version (EN)
- `wechat_url`: Published WeChat URL (if live)
- `linkedin_url`: Published LinkedIn URL (if live)
- `engagement_wechat_views`: WeChat views
- `engagement_wechat_likes`: WeChat likes (Ëµû)

### `lps.json`
Limited partner (investor) database. Array of LP objects.

**Usage**: LP communications, capital call tracking, distribution processing, ILPA reporting.

**Fields**:
- `lp_id`: Unique identifier (e.g., "lp-lightspeed-2026")
- `name`: LP name (individual or institution)
- `entity_type`: Type (Individual, Fund, Family Office, Corporate, Other)
- `commitment_usd`: Total LP commitment to fund
- `called_usd`: Amount called to date
- `called_pct`: Percentage called (called_usd / commitment_usd)
- `last_communication_date`: When we last communicated
- `last_communication_type`: Type (Email, Call, Board meeting, etc.)
- `email`: Primary contact email
- `phone`: Primary contact phone
- `secondary_contact_name`: Backup contact name
- `secondary_contact_email`: Backup contact email
- `status`: Current status (Active, Inactive, Under review)
- `location_country`: LP country
- `notes`: Internal notes (use caution with sensitive info)
- `last_updated`: Last modification date

### `compliance.json`
Regulatory compliance calendar and filing status. Array of compliance items.

**Usage**: Compliance tracking, regulatory deadline alerts, filing status checks.

**Fields**:
- `compliance_id`: Unique identifier (e.g., "comp-form-d-2026-q1")
- `item`: Compliance item name (e.g., "Form D filing", "Annual audited financials", "LP notification")
- `category`: Category (SEC, State, Fund, Tax, LP, Other)
- `deadline`: Due date (ISO 8601)
- `days_until_due`: Auto-calculated days remaining
- `status`: Current status (Not started, In progress, Complete, Overdue)
- `owner`: Team member responsible
- `filing_ref`: Reference doc path or ID
- `notes`: Notes or escalation info
- `completed_date`: When item was completed (if done)
- `last_updated`: Last modification date

### `public_holdings.json`
Public company positions held or tracked for the fund. Array of position objects.

**Usage**: Public market tracking, earnings season prep, peer comps, market research.

**Fields**:
- `position_id`: Unique identifier (e.g., "pos-nvda-2026")
- `ticker`: Stock ticker (e.g., "NVDA")
- `company`: Company name
- `sector`: Sector
- `position_type`: Type (Held, Monitored, Peer comp)
- `shares_held`: Number of shares (if held)
- `cost_basis_per_share`: Cost basis per share
- `current_price`: Latest market price
- `price_update_date`: When price was last updated
- `market_cap_usd`: Latest market cap
- `earnings_date`: Next earnings announcement date
- `latest_eps`: Latest reported EPS
- `pe_ratio`: Latest P/E ratio
- `dividend_yield_pct`: Dividend yield %
- `notes`: Research notes or tracking rationale
- `last_updated`: Last modification date

### `watchlist.json`
Broader company watchlist for market research, sector tracking, and pipeline development. Array of watchlist items.

**Usage**: Market research, sector mapping, competitive intelligence, pipeline sourcing.

**Fields**:
- `watchlist_id`: Unique identifier (e.g., "watch-humane-ai-2026")
- `company`: Company name
- `website`: Company website URL
- `sector`: Primary sector
- `stage_estimated`: Estimated stage (Pre-seed, Seed, Series A, etc.)

## Concurrency and Conflicts

### Single-Writer Assumption
The current system assumes a single active writer (SX in isolation mode). If multiple users edit state:
1. Use git branches and pull requests for state changes
2. Merge conflicts are resolved via git (JSON diffs are readable)
3. Commands should lock-check before modifying (e.g., check git status)

### Partial Updates
Commands should update only the fields they modify:
```json
{
  "deal_id": "deal-acme-20260403",
  "status": "Term Sheet",
  "last_updated": "2026-04-03T14:30:00Z"
}
```

Not rewrite the entire file. Tools like `jq` can perform surgical updates.

## Syncing with XLSX

See `sync-state-to-xlsx.md` for the command specification that regenerates xlsx workbooks from JSON state on-demand or via scheduled tasks.

Workflow:
1. **Update JSON**: Commands write changes to state/\*.json
2. **Commit to git**: `git add state/*.json && git commit -m "..."`
3. **Regenerate XLSX** (optional): Run `sync-state-to-xlsx` command to export to Excel
4. **Commit XLSX** (optional): `git add data/*.xlsx && git commit -m "..."`

## Best Practices

- **Always commit state changes**: Make JSON modifications within git commits for auditability
- **Use ISO 8601 dates**: All dates should be `YYYY-MM-DDTHH:MM:SSZ` (UTC) or `YYYY-MM-DD` for date-only fields
- **Keep notes concise in JSON**: Use `_path` fields to link to longer markdown notes in `/deals/`
- **Validate against schema**: Before committing, validate JSON against `schema.json`
- **Update `last_updated` always**: Every modification should bump `last_updated` timestamp
- **Never expose sensitive LP data**: Anonymize LP names in public research or external mentions

## Tools Integration

### Reading State
- Use `jq` for command-line JSON queries
- Load state in Python/Node.js for processing
- Use `grep` for simple searches

### Writing State
- Use `jq` for updates: `jq '.[] | select(.deal_id == "...") | .status = "..." '`
- Load/modify/save in Python/Node.js for complex changes
- Use git to manage versioning and conflicts

### Validation
- `jq -f schema.json state.json` to validate (if using JSON Schema validator)
- Custom validation scripts in Python/shell

## Examples

### Query: Get all active deals
```bash
jq '.[] | select(.status == "Active")' state/deals.json
```

### Query: Get portfolio companies needing follow-on
```bash
jq '.[] | select(.status == "Follow-on needed")' state/portfolio.json
```

### Update: Mark deal as invested
```bash
jq '(.[] | select(.deal_id == "deal-acme-20260403") | .status) |= "Invested"' state/deals.json > state/deals.json.tmp && mv state/deals.json.tmp state/deals.json
```
